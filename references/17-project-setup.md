# 17 Project Setup：项目初始化协议

## 一句话定位

Project Setup 用于一个项目第一次接入 Kun Coding Router，或项目结构 / 状态混乱时，先建立最小上下文档案。

它解决的问题是：

> 下一次 Codex / AI 新窗口进来时，不用重新猜项目目标、目录责任、运行方式、当前阶段和下一步。

---

## 使用时机

出现以下任一情况时启用：

- 新项目第一次开工；
- 旧项目第一次接入 Kun Coding Router；
- 项目没有 `PROJECT_STATE.md`；
- 用户说“下一次项目开始前，我应该先把文件夹分类好”；
- 用户说“先别写业务，先整理项目结构 / 文档入口”；
- AI 每次新窗口都不知道从哪接；
- 项目启动命令、目录责任、验收方式不明确；
- 项目已经有代码，但缺少状态记录和下一步入口。

---

## 不使用时机

以下情况不要强行启用：

- 用户只是要一个纯文案小改；
- 用户明确说不要整理文档；
- 项目只是一次性实验脚本；
- 当前任务紧急修 Bug，且 Setup 会阻碍最小修复；
- 已有 PROJECT_STATE / CONTEXT / ACCEPTANCE 且内容仍准确。

---

## 核心原则

Project Setup 不是“大整理”，而是建立项目“户口本”。

只做三类事：

1. 识别已有结构；
2. 建立最小上下文文件；
3. 明确下一轮施工入口。

不要在 Setup 阶段写业务代码。

---

## 必须先检查

### 1. 项目基本信息

- 项目名称
- 一句话目标
- 当前阶段
- 当前分支
- 当前启动方式
- 当前技术栈
- 当前部署方式，如有

### 2. 已有文件结构

重点查看：

```text
README.md
PROJECT_STATE.md
HANDOFF.md
AGENTS.md / CLAUDE.md
docs/
src/
app/
pages/
components/
services/
routes/
controllers/
api/
server/
backend/
package.json / pyproject.toml / requirements.txt 等
```

### 3. 风险点

- 是否已经有后端目录但未验收？
- 是否有多个入口文件？
- 是否启动命令不明确？
- 是否 docs 和代码明显不一致？
- 是否有密钥、临时文件、构建产物混入项目？
- 是否 AI 施工规则散落在聊天记录里，没有写进项目？

---

## 推荐建立的最小文件

### 必须：PROJECT_STATE.md

作用：下一轮 AI 接手卡。

必须包含：

- 项目目标
- 当前阶段
- 当前目录 / 分支
- 已完成
- 当前任务
- 下一轮入口
- MVP 范围
- 第一版不做
- 核心验收标准
- 风险 / 待确认
- 后端骨架验收状态
- 最近一次 Git 状态

使用模板：默认用 `PROJECT_STATE.minimal.template.md`（最小版，适合新手和小项目）。当项目变大、需要记 MVP 边界 / 架构快照 / E2E 路径 / 文档同步表时，再升级到 `PROJECT_STATE.template.md`（完整版）。

---

### 推荐：CONTEXT.md

作用：项目术语和目录责任说明。

适合记录：

- 项目里的业务词汇；
- 页面 / 模块含义；
- 数据字段含义；
- 重要目录的责任；
- 用户自己的特殊叫法；
- 哪些概念不能混用。

示例：

```md
# CONTEXT.md

## 项目一句话
个人游戏市场情报中台：每日收集手游、端游、小程序游戏上线、热度、买量测试等信息。

## 核心术语
- 今日新游榜：当天新上线或新进入重点观察池的游戏列表。
- 买量测试：观察疑似投放测试的产品，不等同于正式爆量。
- 热度来源：TapTap、渠道榜、广告素材、社媒讨论等外部线索。

## 目录责任
- app/：页面入口
- components/：可复用 UI 组件
- services/：业务数据处理
- api/：接口路由
- docs/：产品、规格、架构和任务文档
```

---

### 推荐：ACCEPTANCE.md

作用：项目如何启动、测试、验收。

适合记录：

- 安装依赖命令；
- 本地启动命令；
- 测试命令；
- lint / typecheck 命令；
- 核心用户路径；
- 部署后验收路径；
- 常见失败处理。

示例：

```md
# ACCEPTANCE.md

## 本地启动
- 安装：npm install
- 启动：npm run dev
- 访问：http://localhost:3000

## 常用验收
- npm run lint
- npm run typecheck
- npm run test

## 核心路径
1. 打开首页
2. 进入今日新游榜
3. 查看游戏列表
4. 筛选平台
5. 进入详情页
```

---

### 可选：DECISIONS.md

作用：记录不希望后续 AI 反复推翻的关键决策。

适合记录：

- 为什么选这个技术栈；
- 为什么这样划分目录；
- 为什么第一版暂不做某功能；
- 哪些方案被否决；
- 哪些设计后续不能轻易改。

示例：

```md
# DECISIONS.md

## 2026-06-19：第一版先用 mock 数据

原因：
- 当前目标是先跑通仪表盘和榜单体验。
- 自动采集涉及平台规则和稳定性，放到后续阶段。

结论：
- MVP 先做手动 / mock 数据。
- 不在第一版接复杂爬虫。
```

---

### 可选：HANDOFF.md

作用：跨窗口 / 跨会话接续。

当本轮结束、用户准备新窗口继续时生成。

详细规则见：`18-handoff-protocol.md`

---

### 可选：AGENTS.md / CLAUDE.md

作用：AI 施工规则。

只有当项目会频繁由 AI 施工，且存在稳定规则或禁区时才建议创建。

应该放：

- 禁止事项；
- 项目结构速查；
- 常用命令；
- 高风险提醒；
- 必须遵守的业务规则；
- 重要文档指针。

不应该放：

- 历史流水账；
- 单次任务细节；
- 长篇复盘；
- 临时想法。

---

## 执行步骤

### Step 1：扫描项目现状

输出：

```md
## 项目现状扫描

- 项目名称：
- 当前阶段：
- 技术栈：
- 启动命令：
- 已有状态文件：
- 已有 docs：
- 后端目录：有 / 无
- 当前风险：
```

---

### Step 2：判断需要建立哪些文件

输出：

```md
## 建议建立 / 更新

必须：
- [ ] PROJECT_STATE.md

推荐：
- [ ] CONTEXT.md
- [ ] ACCEPTANCE.md

可选：
- [ ] DECISIONS.md
- [ ] HANDOFF.md
- [ ] AGENTS.md / CLAUDE.md

不建议创建：
-
```

---

### Step 3：生成最小内容

只生成下一轮会立刻用到的内容。

不要为了“完整”写一堆空壳文档。

---

### Step 4：确认下一步施工入口

输出：

```md
## 下一轮入口

下一轮建议调用：Kun Coding Router

必须先读：
- PROJECT_STATE.md
- CONTEXT.md（如已创建）
- ACCEPTANCE.md（如已创建）
- HANDOFF.md（如已创建）

建议任务：
-

本轮禁止：
-
```

---

## 最重要原则

Project Setup 的目标不是“看起来很专业”。

目标只有一个：

> 让下一轮 AI 能稳稳接住项目，不乱猜、不乱改、不返工。
