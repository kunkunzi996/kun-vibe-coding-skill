# 12 Verification / Git / Report：验收、保存、汇报

## 真实验收

不要只根据代码阅读就说“完成”。能跑真实路径就必须跑。

### 前端 / UI

- 页面是否能打开。
- 样式是否正常。
- 按钮是否可点击。
- 移动端是否有明显问题。

### 数据功能

- 能否新增。
- 能否读取。
- 能否修改。
- 能否删除。
- 刷新后数据是否还在。

### Bug 修复

- 原 bug 路径是否修好。
- 附近正常流程是否仍然可用。

### 部署

- 本地能否启动。
- build 是否通过。
- 环境变量是否完整。
- 数据是否持久化。
- 公网地址是否可访问。

### 架构调整

- 旧入口是否还能用。
- 路由是否正常。
- API 是否兼容。
- 导入路径是否正确。
- 旧功能是否受影响。

## 无法验收时必须说明

- 哪些验收没跑。
- 为什么没跑。
- 用户应该如何手动检查。

## 开工前 Git 基线

如果项目即将进入首次代码施工，必须先检查：

```bash
git status
```

如果不是 Git 仓库，建议先建立初始存档：

```bash
git init
git add .
git commit -m "init project baseline"
```

对新手解释：

> Git commit 就是代码存档。正式施工前先存一个干净档，后面 AI 改坏了才能回退。

## 用户手动验收指引规则

完成报告里的「你可以这样验收」是必填项，按本轮类型给对应内容：

- 前端 / 页面 / 表单 / 数据功能：给点击路径（打开哪个页面、点哪个按钮、输入什么、保存后看哪里、刷新后数据是否还在）。
- Bug 修复：给原 bug 复现路径 + 修复后验证路径（同样的步骤走一遍，确认问题不再出现）。
- 后端 / API：给接口调用方式、请求示例、预期响应。
- 部署：给公网访问地址、要点开的核心页面、失败时怎么回滚或排查。

例外：

- 轻量小改（UI 小调、文案、单行样式）只给一两步轻量验收，看一眼即可，不用写满全套。
- 当前环境跑不起来时，明确标记「未实际验证」，再给用户照着点的手动步骤，不要假装跑过。

## Git 保存规则

完成修改或验收后，默认不要直接执行 `git add` / `git commit` / `git push`。

默认流程：

1. 先输出修改文件清单、验收结果、当前 `git status`、建议 commit message、是否建议保存版本。
2. 等用户明确确认保存后，再执行 Git 保存命令。

只有在以下情况才允许实际执行 `git add` / `git commit` / `git push`：

- 用户明确说“保存一下 / 提交 Git / 帮我 commit / 同步到 GitHub”。
- 或用户在本轮开始前明确授权“完成并验收后自动提交”。
- 或当前任务本身就是“保存 / Git”。

用户确认后执行：

```bash
git status
git add .
git commit -m "简洁说明本次改动"
git push
```

如果没有远程仓库，只 commit，不假装 push。

如果当前目录不是 Git 仓库，直接说明。

禁止未经用户确认自动 commit / push。

## 与项目洁癖 Gate 的衔接

`12-verification-git-report.md` 只负责验收、Git 和完成报告。

如果本轮满足以下任一条件，应继续启用 `13-project-cleanup-gate.md`：

- 功能完成或阶段完成。
- 部署完成。
- 架构、API、数据模型、目录结构、运行方式发生变化。
- 出现新的 AI 施工禁区或踩坑。
- 准备新开对话，需要下一轮 Codex 接手。
- 用户明确说“收尾 / 同步文档 / 项目状态 / 项目洁癖”。

小 UI、小文案、小样式修改默认不强制进入项目洁癖，除非影响项目说明或后续施工。
