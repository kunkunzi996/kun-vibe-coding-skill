# 09 Test First Gate：测试门

测试门用于逻辑正确性重要的任务。

目标不是增加负担，而是在写代码前先有通过 / 失败标准。

## 使用时机

开启测试门：

- Bug 修复。
- 表单。
- 数据新增 / 读取 / 修改 / 删除。
- API 接口。
- 登录 / 权限。
- 导入 / 导出。
- 统计 / 计算逻辑。
- 数据库变化。
- 复杂业务规则。
- 会影响旧流程的功能。

跳过测试门：

- 纯文案修改。
- 纯样式微调。
- 明确的单行配置修改。

## 最小测试 / 验收清单

```md
# TEST_CHECKLIST

## 正常路径
- 用户正常操作时，应该发生什么？

## 空输入 / 错误输入
- 应该如何拦截、提示或处理？

## 持久化检查
- 刷新、重开、重新加载后数据是否正确？

## 回归检查
- 哪些旧功能不能坏？

## 边界情况
- 最容易漏掉的 1-2 个特殊情况是什么？
```

## Bug 修复规则

原 bug 复现路径就是第一条测试用例。

修复完成的标准：

1. 原 bug 不能再复现。
2. 附近一个正常流程仍然可用。
3. 没有顺手引入无关改动。

## 自动化测试原则

能写自动化测试就先写测试。

不能写自动化测试，就先写手动验收清单。

没有通过测试或验收清单，不能说功能完成。
