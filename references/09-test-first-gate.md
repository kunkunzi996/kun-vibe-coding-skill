# 09 Test First Gate：测试门

测试门用于逻辑正确性重要的任务。

目标不是增加负担，而是在写代码前先有通过 / 失败标准。

## 使用时机

开启测试门：

- Bug 修复。
- 表单。
- 数据新增 / 读取 / 修改 / 删除。
- API 接口。
- 登录 / 权限。
- 导入 / 导出。
- 统计 / 计算逻辑。
- 数据库变化。
- 复杂业务规则。
- 会影响旧流程的功能。

跳过测试门：

- 纯文案修改。
- 纯样式微调。
- 明确的单行配置修改。

## 最小测试 / 验收清单

```md
# TEST_CHECKLIST

## 正常路径
- 用户正常操作时，应该发生什么？

## 空输入 / 错误输入
- 应该如何拦截、提示或处理？

## 持久化检查
- 刷新、重开、重新加载后数据是否正确？

## 回归检查
- 哪些旧功能不能坏？

## 边界情况
- 最容易漏掉的 1-2 个特殊情况是什么？
```

## Bug 修复规则

原 bug 复现路径就是第一条测试用例。

修复完成的标准：

1. 原 bug 不能再复现。
2. 附近一个正常流程仍然可用。
3. 没有顺手引入无关改动。

## 自动化测试原则

能写自动化测试就先写测试。

不能写自动化测试，就先写手动验收清单。

没有通过测试或验收清单，不能说功能完成。

## 对抗式审查：测试门升级模式（可选，上线前 / 定期复盘）

这不是第二套测试门，而是本门的**强度升级档**：上面的清单验证"正常路径 + 常见错误"，对抗式审查验证"恶意 / 极端 / 非正常用户能不能搞崩系统"。功能准备上线或每 2–3 周定期复盘时，让 AI 站在"我要用畸形数据搞崩你"的角度，把整条路径从入口到崩溃走一遍：

- 超大输入（如超大 HTML / 文件）、空值 / 畸形数据；
- 未来时间 / 时区错乱、并发冲突、重试死循环（OOM）、缓存穿透假阳性；
- 边界与异常分支是否都有兜底。

复杂或要上线的功能，如当前工具支持多 Agent / 专项审查就用它并发审查，不支持就按上面清单让当前 AI 手动逐条审查。

这是可选强化，不对小改、不每轮都上；命令示例与完整触发场景见 SKILL.md「两个通用思维」。
