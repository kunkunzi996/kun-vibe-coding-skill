# 18 Handoff Protocol：跨窗口交接协议

## 一句话定位

Handoff Protocol 用于在新窗口、新会话、另一个 AI 或下一轮 Codex 接手前，把当前项目状态压缩成可执行的交接文件。

它解决的问题是：

> 下一轮 AI 不知道上轮做了什么、哪些没做、哪些不能碰，于是重复规划、误改文件、跳过验收或在错误地基上继续施工。

---

## 使用时机

出现以下任一情况时启用：

- 用户说“下次继续”；
- 用户说“新窗口继续”；
- 用户说“先到这里”；
- 用户说“帮我写下一轮 Codex 入口”；
- 本轮任务较复杂；
- 本轮只完成一部分；
- 项目状态发生明显变化；
- 架构、API、数据模型、运行方式发生变化；
- 下一轮需要另一个 AI / Codex 会话接手；
- 准备阶段收尾或长期暂停。

---

## 不使用时机

以下情况不强制生成 Handoff：

- 纯文案小改；
- 纯样式小改；
- 用户明确说不需要交接；
- 本轮没有改变项目状态；
- PROJECT_STATE 已经足够清晰，且没有未完成事项。

---

## Handoff 与 PROJECT_STATE 的区别

### PROJECT_STATE.md

项目长期状态。

记录：

- 当前阶段；
- 已完成；
- MVP 范围；
- 运行 / 验收方式；
- 后端验收状态；
- 下一步。

### HANDOFF.md

下一轮接手说明。

记录：

- 上一轮具体做了什么；
- 下一轮必须先读什么；
- 哪些文件刚改过；
- 哪些坑不能踩；
- 建议调用哪些 Skill；
- 下轮第一步应该做什么。

一句话：

```text
PROJECT_STATE 是项目户口本。
HANDOFF 是下一班交接纸条。
```

---

## HANDOFF.md 必须包含

> V0.7.1：HANDOFF 现在有两版模板，按任务复杂度选用——
> - **最小版** `HANDOFF.minimal.template.md`（4 项核心：本轮目标 / 未完成 / 下一轮必须先读 / 禁止事项）：普通任务默认用这个。
> - **完整版** `HANDOFF.template.md`（下面 11 项全包）：复杂任务、多模块改动、涉及架构/数据/API、或交给另一个 AI 长程接手时用。

完整版必须包含：

1. 本轮目标
2. 已完成内容
3. 修改过的文件
4. 未完成内容
5. 当前风险
6. 下一轮建议任务
7. 下一轮建议调用的 Skill / references
8. 下一轮必须先读的文件
9. 下一轮禁止误做的事情
10. 验收状态
11. Git 状态，如可获取

---

## 生成前必须检查

### 1. 本轮是否已经验收

如果已验收，记录验收命令 / 路径 / 结果。

如果未验收，必须明确写：

```text
验收状态：未完成 / 未执行 / 当前环境无法执行。
```

不要假装已验收。

---

### 2. 本轮是否已经更新 PROJECT_STATE

如果没有更新，需要说明：

- 为什么没有更新；
- 是否建议下一轮先更新；
- Handoff 中哪些内容需要同步回 PROJECT_STATE。

---

### 3. 是否涉及风险

重点检查：

- 数据模型变化；
- API 变化；
- 目录结构变化；
- 新依赖；
- 部署方式变化；
- 未跑测试；
- 后端未验收；
- 用户未确认的产品范围。

## 下一轮入口标准写法

Handoff 末尾必须给下一轮可直接复制的入口。

示例：

```text
下一轮 Codex 入口：

使用 Kun Coding Router 继续当前项目。
请先读取：
1. PROJECT_STATE.md
2. HANDOFF.md
3. docs/TASKS/TASK_002_today_new_games.md
4. app/rankings/page.tsx

本轮任务：继续完成“今日新游榜”的数据接入。

禁止：
- 不要重构整个排行榜模块。
- 不要更换数据源。
- 不要删除 mock 数据。

建议启用：
- Task Spec
- Test First Gate
- Safe Construction
- Verification / Git / Report
```

## Handoff 更新规则

### 可以覆盖旧 HANDOFF

如果旧 Handoff 已经过期，且新一轮完全替代它，可以覆盖。

### 可以追加历史

如果项目处于连续多轮任务中，可以在文件底部追加：

```md
## 历史 Handoff 摘要
- 2026-06-19：完成页面骨架，未接真实数据。
```

但不要让 HANDOFF.md 变成长篇变更日志。

### 长期历史移到 CHANGES.md

如果有多个阶段历史，应迁移到：

- `docs/CHANGES.md`
- `docs/TROUBLESHOOTING.md`
- `docs/TASKS/`

Handoff 只保留下轮马上要用的信息。

---

## 常见错误

### 错误 1：只写“下次继续”

不够。

必须说明继续什么、读哪些文件、禁做什么。

### 错误 2：把 Handoff 写成流水账

不需要记录所有过程。

只保留下轮接手必须知道的信息。

### 错误 3：没有验收状态

下一轮不知道能否基于当前代码继续。

必须明确已验收 / 未验收 / 无法验收。

### 错误 4：遗漏禁止事项

新窗口最容易犯的错是“好心重构”。

必须写清楚哪些不能碰。

## 最重要原则

Handoff 不是为了写得完整。

Handoff 是为了让下一轮 AI 不犯低级错。

宁可短，也要准。
