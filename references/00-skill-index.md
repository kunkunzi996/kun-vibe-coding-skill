## 自然语言触发规则

### 用户说“新项目 / 想法 / 开工”

优先启用：

1. 想法压力测试
2. 开工规格
3. 产品简述
4. 执行规格
5. 必要时设计门 / 架构门

### 用户说“继续开发 / 接着做 / 加功能”

优先启用：

1. PROJECT_STATE
2. 单次任务
3. 测试门（按需）
4. 安全施工
5. 真实用户验收（按需）
6. 保存汇报

### 用户说“修 bug / 不对 / 报错”

优先启用：

1. 单次任务
2. 测试门
3. 安全施工
4. 真实用户验收（按需）
5. 保存汇报

### 用户说“上线 / 部署 / 公网 / 服务器 / Railway / Docker”

优先启用：

1. 单次任务
2. 架构门
3. 安全施工
4. 真实用户验收
5. 保存汇报

### 用户说“收尾 / 同步文档 / 项目状态 / 新开对话前”

优先启用：

1. 保存汇报
2. 项目洁癖

只处理当前代码项目内部的文档和状态同步，不默认处理 Obsidian 或个人知识库。

### 用户说“验收 / 跑通 / 像真实用户一样测”

优先启用：

1. 测试门
2. 真实用户验收
3. 保存汇报

### 用户说“后端搭好了 / 能不能开始写业务 / 验收后端架构 / 后端越写越乱”

优先启用：

1. 后端验收官（13-backend-architecture-acceptance.md）
2. 通过后再走 保存汇报，提交 Git 稳定版本

本门是 07 架构门的下游验收门：07 是开工前画蓝图，13 是骨架搭完后对照蓝图收货。

## 跳过规则

- 已进入研发中期：跳过想法压力测试、开工规格、完整 Product Brief。
- 只是小 UI / 文案：跳过开工规格、执行规格、架构门。
- Bug 修复：跳过产品定位，先写复现路径。
- 只是保存：先检查验收和 Git 状态，不要重新规划。
- 只是小改：不强制项目洁癖，除非影响 README、PROJECT_STATE、架构、API、数据模型、运行方式或 AI 施工规则。
- 项目洁癖：只同步项目内部文档，不默认处理 Obsidian。
- 用户明确说“不要重写 PRD / 不要重做规格”：必须遵守。
